# Version 0.3.0

**Release Date:** March 2026

## Overview

Version 0.3.0 is a major capability release focused on quality gates, compliance integrity, cross-epic coordination, and AI agent reliability. Six epics shipped, covering the full stack from database schema to UI components.

---

## New Features

### Pre-Implementation Pre-Flight Gate (ENG-E217)

A configurable quality gate now blocks implementation dispatch when an epic does not meet minimum standards.

- **Four checks:** scaffold hints present on all tasks, minimum acceptance criteria per feature, minimum task density per feature, epic description quality score (all thresholds configurable via env vars)
- **API:** `GET /epics/:id/preflight` returns pass/fail per check; `POST /epics/:id/preflight-override` records a bypass with a mandatory reason note
- **UI:** `PreflightBadge` on epic detail shows pass/warning/fail state; `PreflightDetailPanel` lists issues; `PreflightOverrideDialog` captures reason before bypass
- **Compliance tab:** Pre-Flight Overrides page logs all bypass events with reason notes and timestamps

### Cross-Epic Dependency Graph (ENG-E220)

Epics can now declare explicit dependencies on other epics, enforced at the dispatch layer.

- **Schema:** `EpicDependency` model — predecessor/successor pairs with optional notes
- **API validation:** `POST /epics/:id/dispatch` blocks dispatch when any declared dependency is still active
- **Preflight integration:** Dependency check included in the pre-flight gate; override available for exceptions
- **UI:** `DependencyBadge` on epic detail shows blocked/clear state; `DependencySelector` for managing dependencies; Pipeline Tracker shows blocked state visually

### Compliance Namespace: Test Artifact Quarantine (ENG-E222)

Laws and compliance records are now namespaced, separating production rules from test artifacts.

- **Schema:** `namespace` field on `Law` model with values `production`, `test`, `archived`
- **API and MCP:** All law queries filter to `production` namespace by default; test laws excluded from Barney and Judge audit runs
- **Dashboard:** Namespace filter in Compliance Laws tab; archived laws hidden by default
- **Migration scripts:** `scripts/tag-test-laws.ts` tags existing junk test laws; `scripts/archive-old-test-laws.ts` archives them; nightly archive cron registered
- **Barney and Judge hardening:** Both auditors skip non-production laws, preventing false positives from test artifacts

### Task Enrichment Layer: Scaffold Hints (ENG-E221)

Tasks now carry `scaffoldHints` — structured file path guidance generated by the planner and surfaced to implementing agents.

- **Schema:** `scaffoldHints` field on `Task` model (JSON array of file path hints)
- **API:** `scaffoldHints` included in task create/update and MCP tool schemas
- **MCP validation gate:** Rejects task creation without scaffold hints when the epic requires enrichment
- **Planner enhancement:** Planner agent now generates scaffold hints per task at planning time
- **Orchestrator injection:** Feature-worker context includes scaffold hints for implementing agents, directly addressing the root cause of LAW-005 violations (missing `relatedFiles`)

### Planner MCP Self-Scoring Loop (ENG-E219)

The planner agent can now score its own output using MCP tools instead of relying on approximated rubric knowledge.

- **MCP tools:** Planner exposes scoring functions via MCP, allowing structured self-evaluation against the quality rubric
- **Score verification:** Planner self-scores after completing the plan; score is stored and surfaced in the Dispatcher UI
- **Threshold enforcement:** Plans scoring below 95/100 trigger a revision loop before finalizing

### Session Monitor DB-State Fix (ENG-E218)

The session monitor task counter now reads directly from database state rather than counting session events.

- **Root cause fixed:** `computeProgressState()` was counting `SESSION_TASK_COMPLETED` events, which could diverge from actual task state in the database
- **Refactor:** `computeProgressState()` split into `computePhaseState()` (event-based, phase tracking only) and a DB-sourced task count path via `computeSessionProgress()`
- **Session summary:** `SessionSummaryPane` updated to display DB-accurate task counts
- **Test coverage:** 16 unit tests + 16 integration tests, all passing

## Improvements

- **Burndown panel:** Now filters to active and completed epics only — planned-but-not-started epics no longer appear in the picker by default
- **Dispatch pipeline status:** Pipeline badge updates to `Building` immediately when the Start Implementation button is clicked — no more silent gap while the agent registers
- **Remote access via Tailscale:** Dispatcher is now accessible remotely via `https://<your-tailscale-hostname>:8443` using `tailscale serve` — see [Remote Access](#remote-access-via-tailscale) below
- **Session end enforcement:** Bobby sessions now call `POST /sessions/:epicId/end` on completion, preventing zombie active sessions in the Sessions tab
- **AI notes on epic completion:** Bobby now writes comprehensive implementation notes to the epic via `PUT /epics/:id` before marking complete

---

## Bug Fixes

- **Session monitor showing wrong task counts** — Fixed by switching from event counting to DB-state queries
- **Burndown defaulting to unstarted epics** — Now defaults to the active epic or most recently completed
- **Pipeline status not updating on dispatch** — Fixed by wiring `PATCH /epic-requests/:id/pipeline` into the webhook receiver's `/dispatch` handler
- **Zombie active sessions** — Bobby session end is now an explicit required step in the implementation prompt
- **Test artifact laws polluting compliance** — 22 junk FT artifact laws tagged and archived via namespace migration

## Remote Access via Tailscale

Dispatcher can be exposed over Tailscale for mobile and remote access without opening firewall ports.

```bash
# Expose Dispatcher on port 8443 via Tailscale HTTPS
tailscale serve --bg --https=8443 http://127.0.0.1:80
```

Once running, access Dispatcher from any device on your Tailscale network:

```
https://<your-machine-name>.<tailnet>.ts.net:8443
```

This uses Tailscale's built-in HTTPS certificate — no manual TLS setup required.

To disable:
```bash
tailscale serve --https=8443 off
```

## Technical Details

- `EpicDependency` Prisma model: predecessor/successor UUIDs, optional notes, createdAt timestamp
- `Law.namespace` Prisma field: enum (`production`, `test`, `archived`), indexed, default `production`
- `Task.scaffoldHints` Prisma field: JSON array of file path strings
- `PreflightOverride` Prisma model: epicId, reason, overriddenBy, createdAt
- New services: `dependencyService.ts`, `preflightService.ts` (expanded), `lawService.ts` (namespace support)
- New scripts: `scripts/tag-test-laws.ts`, `scripts/archive-old-test-laws.ts`
- New UI components: `DependencyBadge`, `DependencySelector`, `DependencyOverrideDialog`, `PreflightBadge`, `PreflightDetailPanel`, `PreflightOverrideDialog`
- Integration test coverage added for dependency enforcement and preflight gate

---

## Breaking Changes

None for end users.

Developers running automated tests that create `Law` records should set `namespace: "test"` on test fixtures to prevent them from appearing in production Barney/Judge audit runs.

## Migration Notes

1. **Rebuild and redeploy** both API and web packages — schema changes require a Prisma migration on first boot
2. **Tag existing test laws** (optional but recommended):
   ```bash
   npx ts-node scripts/tag-test-laws.ts
   ```
3. **Archive old junk laws** (optional):
   ```bash
   npx ts-node scripts/archive-old-test-laws.ts
   ```
4. **Tailscale remote access** (optional): run `tailscale serve --bg --https=8443 http://127.0.0.1:80`

The Docker entrypoint handles schema migrations automatically on container start.
